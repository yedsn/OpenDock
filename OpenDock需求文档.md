# OpenDock 需求文档

## 1. 产品定位

OpenDock 是一个面向开发者和专业软件用户的资源集合与启动工具，用于统一管理目录、网页、命令、文件、应用入口和插件扩展资源，并支持单个打开、批量打开、打开整个集合。

OpenDock 的核心不是传统项目管理，也不是简单书签工具，而是一个「软件资源集合管理器 + 通用启动器 + 插件扩展平台」。

一句话定义：

> OpenDock 用集合的方式管理所有跟软件使用有关的资源，让用户可以把目录、网页、命令、Office 文件、CAD 文件、设计文件、应用入口等内容统一收纳，并按场景快速打开。

## 2. 设计目标

- 管理所有与开发、办公、设计、工程软件相关的资源。
- 用集合的方式组织资源，像收藏夹一样收纳不同类型的内容。
- 支持单个打开、批量打开、打开整个集合。
- 支持不同资源类型使用不同软件、浏览器、终端或插件打开。
- 支持项目场景，但不把项目作为唯一核心。
- 支持未来通过插件机制扩展更多资源类型和打开方式。
- 让用户可以快速恢复某个工作场景，例如开发一个项目、处理一组 Office 文件、打开一批 CAD 图纸、启动某套工具链。

## 3. 核心概念

### 3.1 工作空间

工作空间是 OpenDock 的顶层使用环境，用来承载用户的全部集合、资源、工具配置和插件配置。

第一版可以默认只有一个本地工作空间，后续再支持多个工作空间或同步。

工作空间字段建议：

| 字段 | 说明 |
| --- | --- |
| 工作空间名称 | 如 `我的工作台` |
| 数据存储位置 | 本地配置路径 |
| 默认打开策略 | 可选 |
| 创建时间 | 自动生成 |
| 更新时间 | 自动更新 |

### 3.2 场景

场景是对集合的上层归类，可以理解为「某类工作上下文」。项目只是场景的一种。

场景示例：

- 官网项目
- CAD 出图工作
- 财务报表处理
- Office 文档整理
- 前端开发环境
- 常用工具
- 客户 A 资料

场景字段建议：

| 字段 | 说明 |
| --- | --- |
| 场景名称 | 如 `官网项目`、`CAD 出图工作` |
| 场景类型 | 项目 / 办公 / 设计 / 工程 / 通用 / 自定义 |
| 场景描述 | 可选 |
| 图标 / 颜色 | 可选 |
| 创建时间 | 自动生成 |
| 更新时间 | 自动更新 |

说明：

- 原来的「项目」可以作为 `项目类型场景`。
- 集合可以关联某个场景，也可以不关联任何场景。
- 用户可以根据场景筛选集合。

### 3.3 集合

集合是 OpenDock 最核心的管理单位，类似一个可打开的资源收藏夹。

集合可以收纳同类型资源，也可以收纳混合资源。第一版建议先支持按集合类型管理，后续通过插件支持更灵活的混合集合。

集合类型示例：

| 集合类型 | 示例 | 典型打开方式 |
| --- | --- | --- |
| 目录集合 | `官网-代码目录` | VSCode / Cursor / 文件管理器 / 自定义编辑器 |
| 网页集合 | `官网-本地网页`、`官网-开发环境网页`、`官网-线上环境网页` | Edge / Chrome / 自定义浏览器 |
| 命令集合 | `官网-启动命令` | CMD / PowerShell / Git Bash / 自定义终端 |
| 文件集合 | `合同文档集合` | 系统默认应用 / Office / WPS / 自定义程序 |
| Office 集合 | `月度报表集合` | Word / Excel / PowerPoint / WPS |
| CAD 集合 | `施工图集合` | AutoCAD / 浩辰 CAD / 中望 CAD / 自定义 CAD 软件 |
| 应用集合 | `常用软件` | 启动指定应用程序 |
| 混合集合 | `官网工作台` | 按集合项类型分别打开 |
| 插件集合 | 由插件定义 | 由插件提供打开和管理能力 |

集合字段建议：

| 字段 | 说明 |
| --- | --- |
| 集合名称 | 如 `官网-代码目录`、`月度报表集合` |
| 集合类型 | 目录 / 网页 / 命令 / 文件 / Office / CAD / 应用 / 混合 / 插件 |
| 关联场景 | 可选 |
| 默认打开工具 | 如 VSCode、Chrome、Excel、AutoCAD |
| 打开策略 | 单个打开 / 批量打开 / 全部打开 |
| 排序 | 控制展示顺序 |
| 备注 | 可选 |
| 创建时间 | 自动生成 |
| 更新时间 | 自动更新 |

### 3.4 集合项

集合项是集合里真正被打开的资源。

集合项示例：

| 集合项类型 | 示例 |
| --- | --- |
| 目录 | `E:\code\official-site` |
| 网页 | `https://example.com/admin` |
| 命令 | `npm run dev` |
| Office 文件 | `D:\docs\月度报表.xlsx` |
| CAD 文件 | `D:\cad\施工图.dwg` |
| 普通文件 | `D:\docs\需求文档.md` |
| 应用程序 | `C:\Program Files\Example\app.exe` |
| 插件资源 | 由插件定义，例如数据库连接、远程服务器、API 文档入口 |

集合项字段建议：

| 字段 | 说明 |
| --- | --- |
| 名称 | 如 `官网前端目录`、`线上后台`、`施工图 A` |
| 类型 | 目录 / URL / 命令 / 文件 / 应用 / 插件资源 |
| 内容 | 路径、URL、命令内容、应用路径或插件数据 |
| 工作目录 | 命令类型需要，可选 |
| 打开工具 | 可继承集合默认工具，也可单独指定 |
| 打开参数 | 可选，例如启动参数、命令参数 |
| 排序 | 控制集合内顺序 |
| 备注 | 可选 |

## 4. 默认创建逻辑

OpenDock 不再只围绕「项目」创建默认集合，而是根据场景类型创建默认集合模板。

### 4.1 项目场景默认集合

创建项目类场景 `官网` 时，默认创建：

- `官网-代码目录`
- `官网-本地网页`
- `官网-开发环境网页`
- `官网-线上环境网页`
- `官网-常用命令`

同一个项目类场景下允许存在多个同类型集合。例如网页资源可以按环境拆分为本地网页、开发环境网页、测试环境网页、线上环境网页，避免所有链接混在一个集合里。

用户可以修改、删除或新增集合，例如：

- `官网-测试环境网页`
- `官网-Dev 环境`
- `官网-部署入口`
- `官网-设计稿`
- `官网-接口文档`

### 4.2 办公场景默认集合

创建办公类场景 `月度报表` 时，可以默认创建：

- `月度报表-Excel 文件`
- `月度报表-相关网页`
- `月度报表-常用目录`

### 4.3 CAD 场景默认集合

创建 CAD 类场景 `施工图整理` 时，可以默认创建：

- `施工图整理-CAD 文件`
- `施工图整理-图纸目录`
- `施工图整理-相关资料`

### 4.4 无场景集合

用户可以创建不关联任何场景的集合，例如：

- 常用工具网页
- 常用软件
- 临时文件集合
- 常用命令

无场景集合显示在「全部集合」或「无场景集合」视图中。

## 5. 功能需求

### 5.1 场景管理

必须支持：

- 新建场景。
- 编辑场景。
- 删除场景。
- 查看场景列表。
- 根据场景筛选集合。
- 创建场景时根据场景类型创建默认集合。

建议支持：

- 场景搜索。
- 场景排序。
- 场景归档。
- 场景图标或颜色标记。
- 自定义场景模板。

### 5.2 集合管理

必须支持：

- 新建集合。
- 编辑集合。
- 删除集合。
- 设置集合类型。
- 设置是否关联场景。
- 按场景筛选集合。
- 按集合类型筛选集合。
- 查看无场景集合。
- 查看全部集合。
- 添加、编辑、删除集合项。

集合的主要操作：

- 单个打开集合项。
- 批量打开选中的集合项。
- 打开整个集合。
- 打开某个场景下的全部集合，后续可作为增强功能。

### 5.3 集合项管理

必须支持：

- 新建集合项。
- 编辑集合项。
- 删除集合项。
- 设置集合项类型。
- 设置集合项内容。
- 设置集合项打开工具。
- 设置命令类集合项的工作目录。
- 设置集合项排序。

建议支持：

- 拖拽排序。
- 批量导入路径或 URL。
- 从文件选择器添加文件。
- 从目录选择器添加目录。
- 检测路径是否存在。
- 检测 URL 格式是否有效。

### 5.4 打开方式管理

OpenDock 需要支持不同资源类型使用不同工具打开。

目录资源：

- VSCode。
- Cursor。
- 文件管理器。
- 自定义编辑器。

网页资源：

- Edge。
- Chrome。
- Firefox，可选。
- 自定义浏览器。

命令资源：

- CMD。
- PowerShell。
- Git Bash。
- Windows Terminal，可选。
- 自定义终端。

Office 文件：

- 系统默认应用。
- Microsoft Office。
- WPS。
- 自定义程序。

CAD 文件：

- 系统默认应用。
- AutoCAD。
- 浩辰 CAD。
- 中望 CAD。
- 自定义 CAD 软件。

应用资源：

- 直接启动应用程序。
- 支持启动参数。
- 支持以指定工作目录启动。

自定义工具需要配置：

| 字段 | 说明 |
| --- | --- |
| 工具名称 | 如 `Trae`、`自定义 Chrome`、`AutoCAD 2024` |
| 工具类型 | 编辑器 / 浏览器 / 终端 / Office / CAD / 应用 / 插件 |
| 可执行文件路径 | 如 `C:\Users\...\Code.exe` |
| 启动参数模板 | 如 `{path}`、`{url}`、`/k {command}` |
| 是否默认工具 | 可选 |

### 5.5 插件机制

插件机制是 OpenDock 后续扩展的核心能力，形态上可以参考 Obsidian：核心应用只保留稳定的工作区、场景、集合、资源、打开工具和设置能力，更多资源类型、同步方式、导入方式、专业软件集成通过插件扩展。

插件不是左侧导航里的资源筛选入口，而是放在 `设置 -> 插件管理` 中集中管理。插件启用后，可以向 OpenDock 注册新的集合类型、集合项类型、打开动作、设置面板、后台任务和同步能力。

插件管理应包含三类区域：

- 已安装插件：查看插件名称、能力、版本、状态，支持启用 / 禁用。
- 插件市场 / 插件库：展示可安装插件，第一阶段可以是内置推荐列表，不需要真实在线市场。
- 插件配置：插件启用后提供自己的配置项，例如同步地址、同步间隔、冲突策略、允许访问的数据范围。

插件安装后的配置入口应出现在设置页左侧菜单中，类似 Obsidian 的插件设置：`插件管理` 负责安装、启用、禁用和权限查看；具体插件如 `WebDAV Sync`、`AList Import` 拥有独立设置菜单项。

插件可以扩展：

- 新的集合类型。
- 新的集合项类型。
- 新的打开工具。
- 新的导入方式。
- 新的资源解析方式。
- 新的软件集成能力。
- 新的后台任务，例如定时同步、定时检查、批量导入。
- 新的设置页分组或插件专属配置面板。

插件运行边界建议：

- 插件必须声明 manifest，包括插件 ID、名称、版本、作者、能力、权限、设置 schema。
- 插件权限应明确展示，例如读取工作区、写入工作区、访问网络、访问本地文件、执行命令。
- 涉及写入、删除、执行命令、访问网络同步的插件，需要用户显式启用并确认权限。
- 插件不应直接绕过 OpenDock 的打开工具配置；打开资源时应通过 OpenDock 的工具和打开策略执行。
- 插件产生的数据应归属到当前工作区，便于导出、备份和同步。

插件 manifest 字段建议：

| 字段 | 说明 |
| --- | --- |
| id | 插件唯一 ID，如 `webdav-sync` |
| name | 插件名称，如 `WebDAV Sync` |
| version | 插件版本 |
| author | 作者 |
| description | 插件说明 |
| capabilities | 能力声明，如 `sync`、`collection-type`、`resource-opener` |
| permissions | 权限声明，如 `workspace:read`、`workspace:write`、`network:webdav` |
| settingsSchema | 插件配置 schema |
| backgroundTasks | 后台任务声明，如自动同步 |

### 5.5.1 WebDAV Sync 插件（后续）

WebDAV Sync 是一个典型的同步类插件，用于把 OpenDock 的工作区数据同步到用户自己的 WebDAV 服务，例如 NAS、坚果云、Nextcloud、AList 等。

核心目标：

- 支持用户配置 WebDAV 地址、用户名、密码或 Token、远端目录。
- 支持手动同步和自动同步。
- 支持同步当前工作区的场景、集合、集合项、工具配置、插件配置。
- 支持展示最近同步时间、同步状态、失败原因。
- 支持冲突策略配置，例如本地优先、远端优先、保留两份、手动处理。
- 支持同步前备份，避免配置误覆盖。

WebDAV Sync 配置项建议：

| 配置项 | 说明 |
| --- | --- |
| serverUrl | WebDAV 服务地址 |
| username | 用户名 |
| credential | 密码 / Token，实际产品中应加密存储 |
| remotePath | 远端目录，如 `/OpenDock/workspaces` |
| autoSync | 是否自动同步 |
| syncInterval | 自动同步间隔 |
| syncScope | 同步范围：当前工作区 / 全部工作区 |
| conflictPolicy | 冲突策略：本地优先 / 远端优先 / 保留两份 / 手动处理 |
| lastSyncAt | 最近同步时间 |
| status | idle / syncing / success / failed |

WebDAV Sync 安全边界：

- 原型阶段只做交互模拟，不连接真实 WebDAV。
- 正式实现时密码 / Token 不应明文展示，不应写入普通导出文件。
- 自动同步失败不应阻塞本地打开资源。
- 冲突发生时不应静默覆盖，除非用户明确选择固定冲突策略。
- 插件需要网络访问权限，启用前应展示权限说明。

插件示例：

| 插件 | 能力 |
| --- | --- |
| WebDAV Sync 插件 | 将工作区数据同步到用户自有 WebDAV 服务，支持手动同步、自动同步、冲突策略 |
| Office 插件 | 管理 Word / Excel / PPT 文件，支持用指定 Office 或 WPS 打开 |
| CAD 插件 | 管理 DWG / DXF 等图纸文件，支持指定 CAD 软件打开 |
| Browser 插件 | 管理网页集合，支持多浏览器、多窗口策略 |
| Terminal 插件 | 管理命令集合，支持不同终端和执行策略 |
| Database 插件 | 管理数据库连接入口 |
| Remote 插件 | 管理 SSH、远程桌面、服务器入口 |
| API 文档插件 | 管理接口文档、Postman、Apifox 等入口 |

第一版可以先不实现完整插件系统，但设置页原型应体现插件管理、启用状态、权限说明和 WebDAV Sync 配置方式，数据模型和命名应为插件预留扩展空间。


### 5.6 AI / MCP 友好能力（后续）

AI / MCP 友好能力是 OpenDock 后续规划方向，用于让 AI 助手通过标准化接口理解用户的工作区、场景、集合和资源上下文，从而更准确地辅助开发、运维、办公和专业软件工作。

该能力不属于第一版 MVP，建议在基础资源管理、打开工具配置、插件机制稳定后再实现。

核心目标：

- 让 AI 能读取 OpenDock 中的项目/场景信息，理解当前用户在处理什么工作。
- 让 AI 能查询某个场景下有哪些目录、网页、命令、文件、应用和插件资源。
- 让 AI 能根据集合信息给出更准确的操作建议，例如启动哪个项目、打开哪些网页、运行哪些命令。
- 让 AI 能基于资源上下文生成工作计划、排查步骤、发布检查清单、接口调试建议等。
- 让 AI 能通过 MCP 暴露的只读接口获取上下文，而不是依赖用户反复手动描述项目结构。

建议 MCP 能力边界：

| 能力 | 说明 |
| --- | --- |
| 工作区查询 | 获取当前工作区、工作区列表、默认工作区信息 |
| 场景查询 | 获取场景列表、场景类型、场景描述、场景关联集合 |
| 集合查询 | 获取集合列表、集合类型、默认打开工具、集合项数量 |
| 集合项查询 | 获取目录、URL、命令、文件、应用、插件资源等集合项信息 |
| 当前上下文 | 获取当前选中的场景、集合、集合项和最近打开记录 |
| 工具配置查询 | 获取编辑器、浏览器、终端、Office、CAD 等工具配置摘要 |
| 插件资源查询 | 获取插件声明的资源类型和插件集合摘要 |
| 只读搜索 | 按关键词搜索场景、集合、集合项、工具配置 |

安全边界建议：

- 第一阶段 MCP 只提供只读能力，不允许 AI 直接执行命令、打开程序、删除资源或修改配置。
- 涉及命令集合时，MCP 只返回命令内容和工作目录，不自动执行。
- 涉及本地路径时，可以提供路径，但后续应支持敏感路径脱敏策略。
- 涉及插件数据时，由插件声明哪些字段可以暴露给 AI。
- 任何写入、打开、执行类动作都应由用户确认后再触发。

典型使用场景：

- 用户问 AI：「帮我看看官网项目怎么启动」，AI 可通过 MCP 读取 `官网-代码目录`、`官网-本地网页`、`官网-常用命令`，给出启动步骤。
- 用户问 AI：「这个项目有哪些环境入口」，AI 可读取本地网页、开发环境网页、线上环境网页集合并汇总。
- 用户问 AI：「帮我整理发布前检查清单」，AI 可读取命令集合、线上环境网页、监控入口和接口文档入口。
- 用户问 AI：「我当前工作区有哪些 CAD 资源」，AI 可查询 CAD 集合和图纸文件。
- 用户问 AI：「帮我生成这个工作区的说明文档」，AI 可读取场景、集合、集合项结构并生成说明。

建议后续 MCP 资源/工具设计：

| MCP 项 | 类型 | 说明 |
| --- | --- | --- |
| `opendock://workspaces` | Resource | 当前可见工作区列表 |
| `opendock://scenes` | Resource | 场景列表和摘要 |
| `opendock://scene/{sceneId}` | Resource | 指定场景详情 |
| `opendock://collection/{collectionId}` | Resource | 指定集合详情 |
| `opendock://current-context` | Resource | 当前选中场景、集合、集合项 |
| `search_opendock_resources` | Tool | 按关键词搜索资源，默认只读 |
| `summarize_opendock_scene` | Tool | 汇总指定场景结构，便于 AI 理解项目 |

第一阶段建议只实现 Resource 和只读搜索 Tool；第二阶段再考虑受控写入、打开和执行能力。
### 5.7 筛选与搜索

必须支持：

- 根据场景筛选集合。
- 根据集合类型筛选集合。
- 根据关键词搜索场景、集合、集合项。
- 显示无场景集合。

建议支持：

- 最近打开。
- 常用集合。
- 收藏 / 置顶集合。
- 按工具类型筛选。
- 按资源类型筛选。

## 6. 典型用户流程

### 6.1 创建开发项目场景

用户创建场景 `官网`，场景类型选择 `项目`。

系统自动创建：

- `官网-代码目录`
- `官网-本地网页`
- `官网-开发环境网页`
- `官网-线上环境网页`
- `官网-常用命令`

用户进入 `官网-代码目录`，添加：

- `官网前端目录`
- `官网后端目录`

用户进入 `官网-本地网页`，添加：

- `本地前端地址`
- `本地后台地址`

用户进入 `官网-开发环境网页`，添加：

- `开发环境官网`
- `开发环境后台`

用户进入 `官网-线上环境网页`，添加：

- `线上官网`
- `线上后台`

用户进入 `官网-常用命令`，添加：

- `启动前端`
- `启动后端`
- `拉取代码`
- `构建发布`

### 6.2 打开开发环境

用户筛选场景 `官网`。

点击：

- 打开 `官网-代码目录`：用 VSCode / Cursor 打开所有目录。
- 打开 `官网-本地网页`：用 Chrome / Edge 打开本地开发链接。
- 打开 `官网-开发环境网页`：用 Chrome / Edge 打开开发环境链接。
- 打开 `官网-线上环境网页`：用 Chrome / Edge 打开线上环境链接。
- 打开 `官网-常用命令`：用指定终端打开命令。

### 6.3 创建 Office 文件集合

用户创建场景 `月度报表`，场景类型选择 `办公`。

用户创建集合 `月度报表-Excel 文件`。

里面添加：

- `收入统计.xlsx`
- `成本明细.xlsx`
- `汇总报表.xlsx`

用户点击打开集合后，OpenDock 使用 Excel 或 WPS 打开这些文件。

### 6.4 创建 CAD 文件集合

用户创建场景 `施工图整理`，场景类型选择 `工程`。

用户创建集合 `施工图整理-CAD 文件`。

里面添加：

- `平面图.dwg`
- `立面图.dwg`
- `节点详图.dwg`

用户点击打开集合后，OpenDock 使用指定 CAD 软件打开这些图纸。

### 6.5 创建无场景集合

用户创建集合 `常用工具网页`，不关联任何场景。

里面添加：

- GitHub
- Vercel
- Cloudflare
- 公司后台
- API 文档

该集合始终显示在无场景集合或全部集合里。

## 7. MVP 范围

第一版建议只做这些：

- 场景增删改查。
- 场景支持类型：项目、办公、工程、通用、自定义。
- 创建项目类场景时自动生成默认集合，默认集合可以包含多个网页集合。
- 集合增删改查。
- 集合可关联场景，也可不关联场景。
- 集合项增删改查。
- 支持目录、URL、命令、文件、应用五类集合项。
- 支持按场景筛选集合。
- 支持按集合类型筛选集合。
- 支持单个打开、打开整个集合。
- 支持 VSCode、Cursor、Chrome、Edge、CMD、PowerShell、Git Bash。
- 支持使用系统默认应用打开普通文件。
- 支持配置自定义编辑器、浏览器、终端和应用程序。
- 数据模型为插件机制预留字段。
- 数据模型为后续 AI / MCP 只读上下文能力预留字段。

第一版可以暂不做：

- 完整插件市场。
- 复杂插件 SDK。
- 多人协作。
- 权限管理。
- 云同步。
- 标签系统。
- 复杂仪表盘。
- 任务 / 进度 / 甘特图。
- 自动识别所有文件类型。
- 深度集成 Office 或 CAD 软件内部能力。

## 9. 验收标准

### 9.1 创建项目类场景

Given 用户输入场景名称 `官网`，并选择场景类型 `项目`

When 用户点击创建场景

Then 系统创建场景 `官网`，并自动生成 `官网-代码目录`、`官网-本地网页`、`官网-开发环境网页`、`官网-线上环境网页`、`官网-常用命令` 等默认集合。

### 9.2 修改默认集合

Given 场景 `官网` 已存在

When 用户把 `官网-代码目录` 改名为 `官网-前端目录`

Then 集合名称更新成功，仍然关联 `官网` 场景。

### 9.3 添加目录集合项

Given 用户进入 `官网-代码目录`

When 用户添加目录 `E:\code\official-site`

Then 该目录作为集合项保存，并可用 VSCode、Cursor 或文件管理器打开。

### 9.4 添加网页集合项

Given 用户进入 `官网-开发环境网页`

When 用户添加 URL `https://example.com`

Then 该链接保存成功，并归入对应环境网页集合，可用 Chrome 或 Edge 打开。

### 9.5 添加命令集合项

Given 用户进入 `官网-常用命令`

When 用户添加命令 `npm run dev`，并设置工作目录

Then 该命令保存成功，并可用指定终端执行。

### 9.6 添加普通文件集合项

Given 用户创建集合 `月度报表-Excel 文件`

When 用户添加文件 `D:\docs\月度报表.xlsx`

Then 该文件保存成功，并可使用系统默认应用或指定 Office 工具打开。

### 9.7 添加 CAD 文件集合项

Given 用户创建集合 `施工图整理-CAD 文件`

When 用户添加文件 `D:\cad\施工图.dwg`

Then 该文件保存成功，并可使用系统默认应用或指定 CAD 工具打开。

### 9.8 按场景筛选集合

Given 系统中存在多个场景和无场景集合

When 用户筛选场景 `官网`

Then 页面只显示关联到 `官网` 的集合，同时可通过入口查看无场景集合。

### 9.9 创建无场景集合

Given 用户创建集合时不选择场景

When 用户保存集合

Then 系统创建一个无场景集合，可在全部集合或无场景集合视图中查看。

### 9.10 打开整个集合

Given 集合中存在多个集合项

When 用户点击打开整个集合

Then OpenDock 按集合项类型和配置的打开工具依次打开所有集合项。

## 10. 推荐信息架构

页面建议分为：

- 场景列表。
- 集合列表。
- 集合详情。
- 工具配置。
- 插件配置，后续。
- 最近打开，可选。

主界面建议使用三栏结构：

- 左侧：场景列表，包括全部、无场景、最近、收藏等入口。
- 中间：当前场景 / 全部场景下的集合列表。
- 右侧：选中集合的集合项和打开操作。

这种结构能支持开发项目，也能支持办公文件、CAD 图纸、应用程序和插件资源。
